Dev Server Management

The dev server (CX22) is used for testing new themes and features before production deployment.

Dev User Setup

A dedicated lofidev user isolates testing from the system. Set up once per server:

# Copy scripts to server
scp scripts/*.sh root@5.78.42.22:/tmp/

# SSH in and run setup
ssh root@5.78.42.22
bash /tmp/setup-dev-user.sh
bash /tmp/install-dev-reset.sh

Daily Auto-Reset

The dev server resets daily at 4 AM UTC to keep it clean:

  • Kills all lofidev processes (preserves SSH sessions)
  • Cleans home directory (preserves .ssh, .bashrc)
  • Recreates empty ~/streams/ directory

Manual Reset

# Trigger reset manually
ssh root@5.78.42.22 '/opt/scripts/reset-dev.sh'

# View reset logs
ssh root@5.78.42.22 'cat /var/log/dev-reset.log'

Testing Workflow

Process CPU RAM Notes
Xvfb ~2-4% ~50MB Minimal overhead
Chromium (browser + GPU + renderer) ~30-224% ~500-600MB Varies heavily by theme complexity
FFmpeg (x264 encoding) ~58-67% ~150-750MB Varies by bitrate (1.5-6 Mbps)
Total per stream ~1-2 vCPU ~700-1400MB Quality themes use more

With 5 streams: ~68% CPU, ~4GB RAM on CPX62 (16 vCPU, 32GB).

Theme Performance Analysis

Each theme has different CPU requirements based on animation complexity. Understanding this helps plan capacity.

Per-Stream CPU Breakdown (Dec 2025)

Platform Theme FFmpeg CPU Chromium CPU Total Notes
DLive Space Station ~67% ~224% High Most complex - SVG animations, blur filters, 3 star layers
Kick Arcade ~66% ~35% Medium Highest bitrate (6 Mbps)
Odysee Underwater ~66% ~32% Medium Particle effects
YouTube Night City ~65% ~58% Medium Rain drops, city lights
Twitch Coffee Shop ~58% ~43% Low Simpler animations, steam effects

Note: Chromium CPU >100% indicates multi-threaded rendering across multiple cores.

CPU-Heavy CSS/JS Features

These features significantly increase Chromium CPU usage:

Feature CPU Impact Example
SVG path animation (d: attribute) Very High DLive waveform animation
filter: blur() High DLive aurora, earth glow
Multiple box-shadow animations Medium-High Star fields with 30+ shadows
Web Audio API oscillators Medium 10+ concurrent oscillators
CSS transform animations Low Rain drops, steam (GPU-accelerated)

Optimization Options (If Needed)

If adding more streams, consider these optimizations for high-CPU themes:

  • Remove SVG path animations - replace with CSS transforms or static waveforms
  • Remove or reduce filter: blur() - use pre-blurred images instead
  • Reduce star layers from 3 to 1
  • Slow down animation frequencies (longer durations = fewer repaints)
  • Use CSS will-change hints for GPU acceleration

Why Separate Chromium Instances?

Each stream runs its own Chromium instance rather than tabs in one browser:

Approach Pros Cons
Separate instances (current) Isolation - one crash doesn't kill all streams ~500MB RAM overhead per instance
Single instance, multiple windows Shared memory, less overhead Single point of failure, resource contention

Decision: With RAM at only 13% usage, the isolation benefit outweighs memory savings. Keep separate instances.

Potential Conflicts & Gotchas

CRITICAL: Stream Key Conflicts

Both servers had the same stream keys. When both tried to stream to YouTube, they fought for the same RTMP endpoint causing instability.

Solution: Dev server services are now disabled. Never run production stream keys on dev.

Known Issues & Solutions

Issue Cause Solution
Stream key conflict Same key on multiple servers Disable services on dev, use test keys
Display collision Two Xvfb on same display Unique display numbers (:98, :99, :97...)
Audio routing issues Wrong sink assignment Named sinks, move-sink-input by PID
Chromium session conflict Shared user data dir --user-data-dir=/tmp/chromium-{platform}

Quality vs Quantity Trade-off

With high-quality animated themes (like Space Station), each stream uses more CPU than initially estimated. 5-6 polished streams with unique themes is the sweet spot for this server. Adding more platforms would require either simpler themes or a server upgrade.
